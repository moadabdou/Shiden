\noindent
\begin{minipage}[t]{0.48\textwidth}
    \vspace{0pt}
    {\fontsize{9.5}{11}\selectfont\color{shidenaccent}\textbf{01 // TOPOLOGY}\par}
    \vspace{1.5mm}
    
    {\fontsize{19}{23}\selectfont\color{white}\textbf{Consistent Hash Ring}\par}
    \vspace{2mm}
    \noindent\rule{\linewidth}{0.4pt}
    \vspace{2.5mm}
    
    \normalsize\color{shidentext}
    Modulo-based hashing ($hash(key) \pmod N$) fails in elastic environments because scaling the cluster size $N$ invalidates almost all cache mappings.
    
    Consistent hashing maps both \textbf{servers} and \textbf{data keys} onto a circular $2^{32}-1$ integer ring. A key routes clockwise to the first virtual node (VNode) it encounters.
    
    To achieve uniform key distribution, we scatter multiple VNodes per physical node across the ring:
    \begin{center}
      \vspace{-1.5mm}
      \small$\text{VNode}_{\text{address}} = \text{Murmur3}(\text{node\_id} + \text{"\#"} + i)$
      \vspace{-1.5mm}
    \end{center}
    
    \takeaway{VNode scattering smooths hash space allocation, reducing load variation across nodes to a Coefficient of Variation (CV) of \textbf{2.98\%}.}
\end{minipage}%
\hfill
\begin{minipage}[t]{0.01\textwidth}
    \centering\color{shidenpurple!20}\rule{0.8pt}{2.8in}
\end{minipage}%
\hfill
\begin{minipage}[t]{0.48\textwidth}
    \vspace{0pt}
    {\fontsize{9.5}{11}\selectfont\color{shidenmuted}\textbf{IMPLEMENTATION \& CONFIGURATION}\par}
    \vspace{3.5mm}
    
    \begin{lstlisting}[language=Java]
// VNode registration: Map servers to ring
for (int i = 0; i < vnodes; i++) {
    int hash = murmur3(node.id() + "#" + i);
    ring.put(hash, node);
}

// Clockwise route lookup on sorted array
int idx = Arrays.binarySearch(hashes, keyHash);
if (idx < 0) {
    idx = -idx - 1;
    if (idx >= hashes.length) idx = 0; // Wrap
}
return nodes[idx];
    \end{lstlisting}
    
    \vspace{3.5mm}

    \textcolor{shidenpurple}{$\bullet$ \textbf{Physical Nodes:}} 5 (Node-1 to Node-5)\\
    \textcolor{shidenpurple}{$\bullet$ \textbf{VNodes per Node:}} 150 (Total of 750 tokens)\\
    \textcolor{shidenpurple}{$\bullet$ \textbf{Consistent Hashing CV:}} \textbf{2.98\%} (Uniform balance)\\
    \textcolor{shidenpurple}{$\bullet$ \textbf{Key Count:}} 100,000 simulated keys
\end{minipage}

\noindent
\begin{minipage}[t]{0.48\textwidth}
    \vspace{0pt}
    {\fontsize{9.5}{11}\selectfont\color{shidenerror}\textbf{02 // HASH COLLAPSE}\par}
    \vspace{1.5mm}
    
    {\fontsize{19}{23}\selectfont\color{white}\textbf{JDK hashCode() Defect}\par}
    \vspace{2mm}
    \noindent\rule{\linewidth}{0.4pt}
    \vspace{2.5mm}
    
    \normalsize\color{shidentext}
    Standard JDK \texttt{String.hashCode()} is a polynomial rolling hash:
    \begin{center}
      \vspace{-1.5mm}
      \small$h = \sum_{i=0}^{n-1} s[i] \cdot 31^{n-1-i}$
      \vspace{-1.5mm}
    \end{center}
    Under sequential vnode naming patterns (e.g. \texttt{"node-X\#Y"}), the resulting hash codes cluster inside a tiny \textbf{0.00009\%} range of the 32-bit integer space.
    
    Because the server boundaries fail to scatter across the ring, nearly all simulated keys map outside the clustered range, routing clockwise to the first active node.
    
    \defect{Under our sequential vnode naming scheme, \texttt{String.hashCode()} produced severe clustering, routing 100\% of keys to Node-1 and exploding load variation CV to \textbf{200.00\%}.}
\end{minipage}%

    \vspace{1.5mm}
    
    {\fontsize{19}{23}\selectfont\color{white}\textbf{Scale-up Key Migration}\par}
    \vspace{2mm}
    \noindent\rule{\linewidth}{0.4pt}
    \vspace{2.5mm}
    
    \normalsize\color{shidentext}
    Adding a node to the cluster requires data migration. Modulo hashing forces $N/(N+1) = \mathbf{83.33\%}$ of keys to redistribute. Consistent hashing minimizes logical key movement to the optimal $\mathbf{14.56\%}$.
    
    However, Consistent Hashing (Strategy A) splits keys into $N \times V = 900$ small token range boundaries. This forces the storage engine to perform hundreds of random disk reads and split-compaction writes.
    
    \takeaway{Decoupling the data movement layout from the token ranges is required to prevent random disk I/O bottlenecks.}
\end{minipage}%

    {\fontsize{9.5}{11}\selectfont\color{shidenaccent}\textbf{04 // OPTIMIZATION}\par}
    \vspace{1.5mm}
    
    {\fontsize{19}{23}\selectfont\color{white}\textbf{Fixed Slot Partitioning}\par}
    \vspace{2mm}
    \noindent\rule{\linewidth}{0.4pt}
    \vspace{2.5mm}
    
    \normalsize\color{shidentext}
    Fixed Slot Partitioning introduces an indirection layer that decouples logical routing from physical storage nodes. We map a fixed number of partitions (e.g. $Q = 1024$ slots) to servers.
    
    Client routing is a flat $O(1)$ bitwise mask array lookup.
    
    During scale-up, we compare the slot allocation table to find migrated slots (taking sub-millisecond times), and stream their contiguous slot directories directly, eliminating key-by-key database scans.
    
    \takeaway{Fixed Slot Partitioning achieves flat \textbf{28.8 ns} lookup times and speeds up database rebalancing by \textbf{9x} over Consistent Hashing.}
\end{minipage}%
